Memory & CLAUDE.md

The files Claude reads before you type a word. Get these right and every session starts smarter, more consistent, and cheaper.

PRESIDIO®© Presidio, Inc.
How memory loads

Four scopes, loaded in order.

Read at startup, most specific last. Lower-scoped files override broader ones.

Enterprise/Library/.../ClaudeCode/CLAUDE.mdOrg-wide policy pushed by IT. Read-only to you.
Project<repo>/CLAUDE.mdTeam rules, checked into git. The one you'll edit most.
Folder<repo>/sub/CLAUDE.mdLoads only when Claude touches files in that subtree.
User~/.claude/CLAUDE.mdYour personal prefs across every project.
Local override<repo>/CLAUDE.local.mdPersonal, git-ignored. Deprecated — prefer imports.
Check what's loaded: run /memory to see every file in effect and jump straight into editing it.
Source: code.claude.com/docs/en/memory
PRESIDIO®© Presidio, Inc.
Writing good memory

Treat it like a README for Claude.

Short and specific — what a new teammate needs on day one. Every line is re-read every turn, so make each one count.

Put in

  • Build / test / lint commands that actually work here
  • Architecture in two lines — where things live
  • Conventions: naming, imports, formatting quirks
  • Gotchas: "don't touch X", "migrations need Y"

Leave out

  • Long prose Claude can read from the code itself
  • Entire style guides — link or import them instead
  • Secrets, tokens, anything you wouldn't commit
  • Stale notes — wrong memory is worse than none
Rule of thumb: if a line wouldn't change what Claude does, it's costing you tokens for nothing.
Source: code.claude.com/docs/en/memory

Scaling memory

Split it up before it bloats.

A bloated CLAUDE.md costs tokens every turn. Use imports and path-scoped rules to pull in detail only where it's needed.

Imports — @path pulls a file in
# CLAUDE.md
See @docs/architecture.md for the data model.
Coding standards: @~/.claude/my-style.md

Imports resolve recursively (max 5 hops). Keeps the root file a thin index that points to detail.

Path-scoped rules — load by location
# .claude/rules/frontend.md
---
paths: ["src/web/**"]
---
Use the design-system components, never raw CSS.

Rules in .claude/rules/ attach only when Claude works on matching files — zero cost elsewhere.

Skills

Anatomy of a skill

---
name: pdf
description: Extract, merge, split PDFs.

Steps, scripts, and references go here…
Description = the trigger. Claude reads only the name + description until it decides the skill applies, then loads the body.

Subagents

The delegation instinct

Your thread"Find every caller of chargeCard() and tell me which skip validation."
→
Subagent contextgrep · open 14 files · read 40K tokens · trace calls · rule out 9 …
40K tokens of noise stays in the subagent → you get a 5-line answer

Delegate when

  • Wide search across many files
  • Throwaway investigation / spike
  • Parallel independent tasks

Don't bother when

  • You already know the file
  • One quick edit
  • You need the intermediate detail yourself

MCP vs built-in tools

MCP vs built-in tools

Built-ins are fast and zero-config. MCP connects Claude to outside systems — Jira, Slack, your internal APIs.

Built-in tools

  • File ops, shell, search — zero config
  • Lowest latency, no token overhead for schemas
  • Use for everything inside the repo

MCP servers

  • Bridge to external apps & data
  • Each tool's schema costs context — add only what you need
  • Add via claude mcp add or a plugin
Every connected MCP tool loads its schema into context. Twenty idle tools means twenty schemas per turn — connect only what the task needs.

Workflow

How to drive Claude on real work — plan before you build, manage context deliberately, and parallelize when it pays.

PRESIDIO®© Presidio, Inc.
The core loop

Explore → Plan → Execute → Verify.

Most failures skip planning. A plan catches wrong assumptions before any code is written.

01
Explore
Read the relevant code first — delegate the wide search to an Explore subagent.
02
Plan
Have Claude lay out the steps. Review and correct before any edit lands.
03
Execute
Implement against the agreed plan, in small reviewable chunks.
04
Verify
Tests, a diff read, or a fresh subagent to check the work.
A 30-second plan review beats a 30-minute debug. The cheapest fix is the one made before code is written.
Source: anthropic.com/engineering/claude-code-best-practices

Managing a full window

Compact, clear, or carry on.

As the window fills, quality drops. Three options.

/clear

Wipe the slate between unrelated tasks. Cheapest, cleanest reset — use it liberally.

/compact

Summarize the thread so far, keep going on the same task. Auto-fires near ~95% full.

Handoff

Write state to a file, start fresh, read it back. Survives a full reset with intent intact.

If Claude starts forgetting earlier instructions or repeating itself, the window is saturated. Don't push through — /clear or compact.

The handoff pattern

The handoff pattern

For work that spans sessions, write progress to a file and rehydrate in a fresh window.

Session A — nearly full"Write what's done, what's left, and key decisions to PROGRESS.md."
→
Session B — fresh"Read PROGRESS.md and continue from where we left off."

What to capture

  • Goal & current step
  • Decisions + rationale
  • What's verified vs pending
  • Open questions / blockers

Why it works

  • Full clean window each session
  • Auditable record of the work
  • Anyone (or any agent) can resume

Parallel work

Git worktrees — parallel sessions, no conflicts.

A separate checkout per branch. Independent Claude sessions that can't step on each other.

git worktree add ../feat-auth feat-auth
git worktree add ../feat-billing feat-billing
# one Claude per folder, fully isolated

Custom subagents can self-isolate with isolation: worktree — changes land on a branch and merge back only if you keep them.

Good for

  • Two features at once, zero merge chaos
  • A risky spike beside the stable branch
  • Running tests on one while coding another
  • Throwaway experiments that auto-clean

Block 04

Advisor & token efficiency

The same habits that make Claude accurate make it cheap. Pick the right model, cache what repeats, and watch where the tokens go.

PRESIDIO®© Presidio, Inc.
The big idea

Accuracy and cost aren't a trade-off.

A bloated context makes Claude worse and costs more. The fix for both is the same.

Sloppy context

  • Everything dumped in, nothing pruned
  • Adherence drops as the window fills
  • You re-pay for the same junk every turn
  • Slower, pricier, more wrong

Deliberate context

  • Only what the task needs, in the room
  • Subagents absorb the noise
  • Caching reuses the stable prefix
  • Faster, cheaper, more right
Token efficiency isn't penny-pinching. It's the same discipline that makes the output better.

Match the model to the job.

Opus for hard problems, Sonnet for most work, Haiku for simple tasks. Overspending on model also slows things down.

Caching & thinking

Pay once for the stable stuff.

An unchanged prefix — system prompt, CLAUDE.md, big files — gets reused at a fraction of full cost. Keep it stable.

Caching wins

  • Cache reads are a fraction of full input cost
  • Stable CLAUDE.md + early file reads get reused
  • Reordering context constantly breaks the cache — don't churn the prefix

Extended thinking

  • "think" / "think hard" → more reasoning budget
  • Worth it on hard problems; wasteful on simple ones
  • Thinking tokens are billed — spend them where they pay off
Don't churn the prefix. Rewriting early context re-bills everything downstream. Add at the end; keep the head stable.

Output density

Output density

Output is expensive. A compact format returns the same information in fewer tokens.

Verbose ask
"Give me a nicely formatted write-up of each result with headings, explanations, and a table…"
Dense ask
"Return just a compact table: id, status, owner. No prose."

Principles

  • Specify the format — don't let it pad
  • "No preamble / no summary" trims output
  • Structured > prose when you'll parse it
  • Shorter output is cheaper and faster
You can't always shrink input, but you can almost always control the output format. Use it.
